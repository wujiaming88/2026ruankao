## 二、项目背景（痛点）

原系统采用 **Python Django 单体架构**，所有功能模块（用户、商品、订单、支付等）集中在一个工程中。随着业务增长，暴露三大问题：

1. **性能瓶颈**：单体应用无法水平扩展，高峰期响应延迟严重
2. **变更影响面大**：改一个模块可能影响其他模块，每次上线需全量回归
3. **迭代周期长**：多人协作同一代码库冲突频繁，发布节奏受限

---

## 三、架构方案

### 3.1 服务拆分

将单体拆分为 **3个核心微服务**：

| 微服务 | 职责 | 说明 |
|--------|------|------|
| **认证服务** | 登录、注册、Token管理 | 独立鉴权，JWT + 统一网关 |
| **用户服务** | 用户信息、地址、偏好 | 读多写少，适合缓存优化 |
| **业务服务** | 商品、订单、交易核心逻辑 | 高并发，需水平扩展 |

### 3.2 技术栈

| 项目 | 选型 |
|------|------|
| **语言** | Go（高性能、协程并发） |
| **框架** | Kite（字节跳动内部RPC框架） |
| **服务通信** | RPC（Thrift协议） |
| **服务发现** | Consul / ETCD（论文可选） |
| **数据库** | MySQL（按服务拆库） |
| **缓存** | Redis |
| **消息队列** | Kafka / RabbitMQ（论文可选） |
| **网关** | API Gateway 统一入口 |

### 3.3 选型理由（论文用）

- 选Go替换Python：**并发能力强**（goroutine比线程更轻量），**编译型语言性能更高**
- 选微服务替换单体：**独立部署**减少变更影响，**独立扩展**解决性能瓶颈，**团队自治**加速迭代
- 选RPC替代RESTful：**内部服务间调用性能更高**，序列化效率优于JSON

## 四、关键技术实践（论文核心段素材）

### ⭐ 要点1：服务拆分策略

- 按照 **DDD（领域驱动设计）** 思想划分限界上下文（Bounded Context）
- 认证是横切关注点，独立为基础服务
- 用户和业务属于不同领域，数据模型差异大，适合拆分
- 拆分粒度：**不宜太细**（5人团队维护3个服务刚好，太多反而增加运维负担）

### ⭐ 要点2：数据库拆分

- 单体时期共享一个 MySQL 数据库
- 微服务后**每个服务独立数据库**（Database per Service）
- 难点：拆库后跨服务查询需要通过 **API聚合** 而非直接 JOIN
- 数据迁移采用**双写 + 数据校验 + 灰度切换**保证平滑过渡

### ⭐ 要点3：缓存一致性

- 用户信息和商品信息使用 **Redis 缓存** 减轻数据库压力
- 问题：缓存与数据库不一致导致用户看到旧数据
- 解决方案：采用 **Cache Aside 模式**（先更新DB，再删缓存）+ **消息队列异步补偿**保证最终一致性
- 热点数据设置合理 TTL，避免缓存雪崩

## 五、项目成果（论文结尾用）

- 系统平均响应时间从 **800ms 降至 200ms**（提升75%）
- 发布周期从 **2周/次 缩短到 3天/次**
- 核心业务服务可独立水平扩展，支撑大促流量
- 团队协作效率明显提升，各服务独立开发、独立测试

---

## 六、不足与改进（论文必写）

1. **监控体系不完善**：微服务拆分后缺少统一的链路追踪（如 Jaeger/Zipkin），问题排查困难。后续计划引入分布式链路追踪。
2. **容器化未落地**：当时未采用 Docker/K8s 部署，部署运维仍有手工环节。后续计划引入容器编排，实现 CI/CD 自动化。
